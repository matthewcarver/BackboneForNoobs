Why Backbone.js? What is the problem?

  • Modern web applications, shockingly (sarcasm) are becoming just that, first class applications that just happen to run in your web browser.
  • Using the old way of writing js to build complex applications will result in a big ball of spaghetti. Javascript files thousands of lines long, multiple levels of nested functions where each function is hundreds of lines long.
  • This is a perfect candidate for using design principles like GoF and SOLID. These patterns may be 30 years old, but they aren't any less applicable. With judicious use of these principles we can build maintainable, complex web applications in javascript (look ma no higher level abstraction needed)
  • Using those principles we can build applications like Document Cloud, Basecamp Mobile, Groupon Now, all the way to highly interactive games.

Why Backbone.js? Why choose Backbone.js?

  • Backbone.js roots are the same as those of all MV* frameworks, it is not strictly MVC (left) or MVP (right).
  • It is best to think of Backbone.js as a MV* framework, while it shares the same roots of MVC/MVVM/MVP/MVWhatever and you can easily draw parallels (e.g., backbone view == presenter), trying to strictly apply these implementations when working with Backbone.js is not the best idea.
  • Backbone.js is a lightweight, flexible framework with an active community with a lot of good resources to learn the ins and outs.
    • On github
    • 5.3kb packed and gziped
    • Only hard dependency is underscore.js
    • Documented well by the authors.

Backbone.js, what are you made of?

  • Backbone.Model

    The model is where the data goes (obviously). It is also where we put conversions, validation, access control, computed properties. The model can define the url where the resource is located on the server. Model properties should be accessed with get() and set() functions, this is so that Backbone can propogate data change events.

  • Backbone.View

    The view is the 'glue' component. It glues a model to a DOM element on the page, glues events to functions, they don't define anything about the HTML or CSS that will be rendered on the page. A single page can have multiple views bound to model changes and events, updating a part of the page independently without having to re-draw the entire page. *WARNING: terminology may be confusing* the Backbone view can use any view/templating engine with the model to render html content.

  • Backbone.Collection

    The collection is an ordered set of models. Similar to the Backbone model, the collection can define where the resource is located on the server. Events that occur on an item in the collection is also fired at the collection level.

  • Backbone.Router

    The router is where we can link hash fragments to actions and events. This allows a Backbone application to have linkable urls that can be bookmarked or shared. Some global and initialization of the actions/events can be placed in the router.

Backbone.js, what are you made of?

  • Backbone.Events

    Events is a module that can be used in any place to bind and trigger custom events.

  • Backbone.History

    History keeps track of the Router hash changes, so that we don't break the internet, huzzah!

  • Backbone.Sync

    This is the function that is called when Backbone attempts to read or save a model/collection from the server. This can be overriden to use any persistence mechanism you need.

  • Backbone.Utility

    A couple utility functions.

    • noConflict can be used to embed a Backbone application on a page without clobbering the pages Backbone.
    • setDomLibrary tells Backbone to use a specific DOM/Ajax library.

  • For an up to date and more detailed explanation, as always go to the source.
